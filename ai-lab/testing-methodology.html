Testing
methodology.

How a model earns—or loses—a place in a real workflow on real hardware.

◎
The core rule: benchmark scores are not enough. Tests use actual tasks, actual tools, and the hardware available at the desk. New to the vocabulary? Start with Local AI Basics.

General conversation

Can it follow instructions, preserve context, explain clearly, and help without filling the answer with noise?

Write a detailed explanation of how a space shuttle is able to take off, orbit, then return to Earth.

• Use approximately 1000 words.
• Use headings and bullet points.
  • Summarize a long document
  • Explain a concept simply
  • Follow formatting constraints
  • Reason through a problem

Coding ability

Can it understand an existing project, modify code safely, diagnose an error, and match the project’s existing style?

Create a Roblox Lua server script that randomly spawns 25 trees in a 200×200 area.

Use a Tree model from ReplicatedStorage, create GeneratedFlora when missing, raycast to the ground, and vary rotation and scale. Return only the commented script.
  • Read a codebase
  • Add a feature safely
  • Fix without regressions
  • Maintain local style

Agent behavior

Can it use tools, create and update files, inspect its own result, and finish a multi-step instruction without losing the thread?

Create utils.js with a date-formatting function. Then update index.html to use it.
  • Use tools correctly
  • Sequence dependent steps
  • React to command output
  • Stop when complete

Hardware reality

A model is not useful locally if it does not fit, runs too slowly, collapses at the desired context length, or becomes unstable under load.

Reality check: the model worked at 2K context, slowed at 8K, and produced unusable output at 16K.
  • Fit inside 4GB VRAM
  • Remain interactive
  • Handle useful context
  • Recover without crashes

The bottom line

Test with real tasks. Measure real performance. Document what actually happens.
